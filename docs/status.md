### Phase 49 — Graph-native demo: curated BODS subgraph pipeline, Neo4j CSV export, slide viewer, licensing

An eight-phase de-risking sequence (tracked on the Notion ticket "Move toward a graph-native internal architecture") built out a reproducible, talk-ready demo graph for OpenCheck using curated BODS v0.4 data from Open Ownership's published pipelines.

**Demo set (Phase 0).** Nine anchor entities were chosen to showcase the full range of OpenCheck risk signals and BODS features: Daily Mail and General Trust (complex corporate structure + trust/arrangement), BP P.L.C. (complex ownership layers + non-EU + trust), Rosneft Deutschland GmbH (sanctioned), Bank Saderat PLC (sanctioned + non-EU), Biffa PLC and Hornsea 1 Limited (complex ownership layers + non-EU), Care UK Social Care Limited (private equity layers), Taqa Bratani Limited (non-EU + related-sanctioned — Abu Dhabi parent), Newcastle United Football Company Limited (non-EU — PIF ownership chain). Each is present in both the GLEIF and UK PSC BODS bulk datasets published by Open Ownership.

**Data sourcing & extraction (Phases 1–3).** Rather than re-running Open Ownership's heavyweight AWS/Kinesis pipelines (UK PSC needs >20 GB RAM / ~14.5 h; GLEIF ingest >8 GB; both AGPL-3.0 and incompatible with OpenCheck's MIT codebase), the phase consumes the published BODS output directly from local SQLite copies (`gleif_version_0_4.db` / `uk_version_0_4.db`). The existing `extract_bods_subgraphs.py` BFS walker was extended with a `--max-hops` hard cap and memory profiling (`scripts/profile_memory.py`). Worst-case peak RSS across all 17 anchor/source combinations: **20.5 MB** — well under the 512 MB Render budget.

**Build pipeline (Phase 5).** `backend/scripts/build_demo.py` encodes the nine anchor LEIs, runs extraction for each against both SQLite databases, merges GLEIF and UK PSC BODS bundles per entity (deduplication by `statementId`), applies `_normalize_bundle()` to heal BODS 0.3-era serialisation artifacts (comma-joined `source.type`, bare `addresses[].country` strings, missing `isComponent` inside `recordDetails`, legacy `describedByEntityStatement` wrappers), and writes per-entity JSON-Lines to `data/demo/` with a `manifest.json`. `make build-demo` is the single reproducible build step.

**Validation (Phase 6).** Step 3 of `build_demo.py` runs both OpenCheck's own `validate_shape()` (boundary edges, interest-type codelist, dangling-reference check) and `lib-cove-bods` (full BODS v0.4 JSON schema + additional checks). All 9 combined bundles and 19 individual cache files pass clean: 0 dangling edges, 0 duplicate `recordId`s, 0 schema errors.

**Cross-source identity verification (Phase 4).** All eight UK+GLEIF entity pairs were hand-verified against the live GLEIF API (filtering by `registeredAs` + `jurisdiction=GB`). Every Companies House number resolves to exactly one GLEIF record with an exact `registeredAs`/`validatedAs` match. No false links.

**Neo4j CSV export (Phase 7 — first deliverable).** `backend/scripts/export_neo4j.py` combines all nine entity JSONL files (deduplicating 4 cross-entity statement overlaps), writes `data/demo/all_demo.jsonl` (1,204 unique statements), then calls `bods-neo4j to-csv` to produce `data/demo/neo4j/` — 10 CSVs (entity, person, identifier, address, country + typed edge files) plus `import.cypher` and `import.sh`. Loaded into Neo4j 5 Community via Docker: **1,468 nodes, 2,887 relationships** (520 Entity, 18 Person, 630 Identifier, 269 Address, 31 Country; 901 CONTROLS, 355 OWNS, 695 HAS_ADDRESS edges). UBO traversal, effective-ownership percentage chains, and circular ownership detection all verified working with the Cypher queries from `bods-neo4j`. `make export-neo4j` regenerates the export.

**Slide-ready HTML viewer (Phase 7 — second deliverable).** `backend/scripts/gen_slide_html.py` converts all nine demo BODS bundles to Cytoscape.js / dagre ownership graphs embedded in a single self-contained HTML file (`data/demo/slides/opencheck_demo.html`, ~1 MB). All JS libraries (Cytoscape 3.28.1, Dagre 0.8.5, cytoscape-dagre 2.5.0) are inlined at generation time so the file works offline at a conference. Edge colours match `BODSGraph.tsx`: OWNS = blue `#1565c0`, CONTROLS = orange `#e65100`, MANAGES = purple dashed `#6a1b9a`, IS_PARTY_TO = teal `#0d9488`, UNKNOWN = grey. Sidebar entity picker, arrow-key navigation, print CSS (one entity per page). `make slides` regenerates it.

**Licensing & AuraDB deferral (Phase 8).** UK PSC data is Open Government Licence v3.0; GLEIF L1+L2 data is CC0 1.0. The combined demo graph is freely shareable in talks and derivative works. AGPL-3.0 pipeline code is not included in OpenCheck. Full attribution wording in `data/demo/LICENCES.md`. Hosted Neo4j (AuraDB) and all-entity-scale graph DBs are explicitly parked with a named revisit trigger: revisit when multi-hop UBO traversal in the live `/lookup` flow measurably exceeds 2 s median latency with DuckDB on the full BODS dataset, or when the demo set grows beyond ~200 anchor entities. Until then, the stack stays SQLite → BODS JSON-Lines → Neo4j Docker for demos only. Decision recorded in `CLAUDE.md`.
